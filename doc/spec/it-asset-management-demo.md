## IT資産・端末管理デモプロジェクト開発プラン

### 目的

このデモプロジェクトは、**tanoshimi.dev の小規模業務システム例**として、**Python / Django を中心にした実装パターン**を見せることを目的にします。

題材は、**社内で利用する IT資産・端末管理システム**です。ノートPC、スマートフォン、モニター、周辺機器などの貸出、返却、在庫、故障、廃棄、棚卸しを管理します。

見せたいポイントは次の3つです。

1. Django の **モデル、Admin、フォーム、一覧画面**を活かして業務システムを素早く立ち上げられること
2. 小規模システムでも、**申請、承認、状態遷移、履歴管理**をきれいに表現できること
3. tanoshimi.dev の他デモと並べても、**業務テーマと技術テーマが重複しにくいこと**

### 題材概要

プロジェクト名の仮称は、**IT Asset Manager** です。

想定する利用者は次の3者です。

| 利用者 | 主な役割 |
| --- | --- |
| 一般社員 | 端末の貸出申請、利用中端末の確認、返却申請を行う |
| 情シス担当者 | 資産登録、貸出承認、故障受付、棚卸しを行う |
| 管理者 | マスタ管理、監査履歴確認、運用設定を行う |

### デモで扱う業務フロー

最小構成の主業務フローは次の通りです。

1. 情シスが端末を資産台帳へ登録する
2. 社員が貸出申請を行う
3. 情シスが承認し、端末を貸出中にする
4. 社員が返却申請を行う
5. 情シスが返却確認し、状態を在庫へ戻す
6. 故障や紛失があれば記録する
7. 棚卸し結果や操作履歴を残す

この流れにより、単純な CRUD だけでなく、**貸出状態、利用履歴、故障管理、棚卸し**まで含んだ業務アプリになります。

### アーキテクチャ方針

小規模デモとして、**Django を中心にしたモノリシック構成**にします。複雑な分散構成にはせず、Django が得意な領域を素直に活かします。

| レイヤー | 採用案 | 役割 |
| --- | --- | --- |
| Frontend | Django Templates + HTMX または最小限の JavaScript | 一覧、詳細、申請、管理画面 |
| Backend | Django | 業務ロジック、フォーム処理、認可、状態遷移 |
| DB | PostgreSQL | 資産、貸出、棚卸し、履歴の永続化 |
| Admin | Django Admin | 管理者向け台帳、マスタ、棚卸し管理 |
| Local Dev | Docker Compose | web / db をまとめて起動 |

### Python / Django アーキテクチャ

このプロジェクトは、**Python + Django によるサーバーサイド中心の業務アプリ**として構成します。利用者向け画面、管理画面、業務ルール、認可、監査対象の更新処理を Django 側に集約し、フロントエンドには重い責務を持たせません。

#### 採用スタック

| 項目 | 採用方針 |
| --- | --- |
| 言語 | Python |
| Webフレームワーク | Django |
| テンプレート | Django Templates |
| 画面補助 | HTMX または最小限の JavaScript |
| DB | PostgreSQL |
| 認証連携 | tanoshimi.dev ポータルからの認証引継ぎ |
| 管理UI | Django Admin |
| ローカル実行 | Docker Compose |

#### 構成の考え方

1. **画面表示**は Django Template を基本とし、一覧更新や一部フォーム補助だけを HTMX または軽量な JavaScript で補います
2. **業務ルール**は Django の View / Form / Service 相当の層に置き、貸出可否、返却可否、状態遷移、棚卸し差異判定をサーバー側で完結させます
3. **永続化**は PostgreSQL に集約し、資産台帳、申請、履歴、棚卸し結果をリレーショナルに管理します
4. **管理導線**は Django Admin を活用しつつ、利用者向けの主要導線は別画面として実装します
5. **認証**は tanoshimi.dev ポータルの認証状態を前提にし、このアプリ単独のログイン画面は持ちません

#### リクエスト処理イメージ

`Portal Auth` → `Django Middleware / Session` → `View` → `Form / Service` → `Model` → `PostgreSQL`

- ポータルから引き継いだ利用者情報をもとに Django 側でセッションを確立する
- View では認可、対象データ取得、フォーム処理開始を担当する
- Form または Service 相当の処理で入力検証と業務ルール判定を行う
- Model では状態遷移に必要なデータ整合性を維持する
- 重要操作は監査ログに記録する

#### Django プロジェクト内部の責務分割

| レイヤー / 配置 | 主な責務 |
| --- | --- |
| `config` | settings、URL ルート、WSGI/ASGI、環境変数設定 |
| `accounts` | ポータル連携済み利用者情報、ロール、セッション補助 |
| `assets` | 資産台帳、カテゴリ、状態、シリアル番号 |
| `loans` | 貸出申請、承認、貸出確定、返却処理 |
| `incidents` | 故障、紛失、廃棄の記録 |
| `inventory` | 棚卸しセッション、実査結果、差異管理 |
| `auditlogs` | 操作履歴、状態変更履歴 |

#### 実装ルール

1. テンプレートに業務判定を書きすぎず、条件分岐は Python 側に寄せる
2. Model に置くべき整合性ルールと、View/Form 側に置くべき画面入力ルールを分離する
3. 状態遷移は ad-hoc に更新せず、メソッドまたは専用処理にまとめる
4. 監査対象操作は更新処理と同じトランザクション単位で記録する
5. ポータル認証の信頼判定は Django サーバー側で行い、フロントエンドで `portal_token` を扱わない

### Django 採用理由

この題材では、Django の次の強みがそのまま効きます。

1. Model で業務データ構造を定義しやすい

5. 権限や監査対象をサーバー側に寄せやすい

### この題材が向いている理由

この題材は、**Django の得意分野である管理画面、モデル設計、一覧運用、フォーム処理**をそのまま見せやすいです。

また、FERMS の予約系とは異なり、**在庫状態、貸出履歴、故障、棚卸し**が主題になるため、tanoshimi.dev のデモ群としても役割が分かれます。小規模ながら業務システムらしさがあり、Django の採用理由も説明しやすい題材です。
