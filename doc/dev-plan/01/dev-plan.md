# 01. Step by Step 開発計画

## 1. 目的

このフェーズでは、`demo03_am` を **Python / Django ベースの IT資産・端末管理デモ** として段階的に立ち上げるための実装順を定義します。

最初の実演ゴールは、計画書どおり **「社員がノートPCを貸出申請し、情シスが承認し、返却までを管理する」** を成立させることです。

## 2. 前提

- アーキテクチャは Django 中心のモノリシック構成
- DB は PostgreSQL
- 利用者向け画面は Django Templates を基本とする
- 管理導線は Django Admin を活用する
- 認証は tanoshimi.dev ポータルから引き継ぐ
- アプリ単独のログイン画面は作らない
- 貸出可否や状態遷移などの業務ルールは Django 側で最終判定する

## 3. 実装ゴール

このフェーズの完了時点で、最低限次を満たす状態を目指します。

1. Django プロジェクトが起動できる
2. PostgreSQL と接続できる
3. 資産台帳を登録・参照できる
4. 貸出申請、承認、返却の基本フローが動く
5. 主要な状態変更が監査ログに残る
6. デモ用データで一連のシナリオを再現できる

## 5. Step by Step

### Step 1. プロジェクト基盤を作る

**目的:** Django アプリを実装できる最小土台を整える。

**やること**

1. Django プロジェクトを初期化する
2. `config` 配下に settings、URL ルート、WSGI/ASGI を整える
3. PostgreSQL 接続設定を環境変数ベースで定義する
4. Docker Compose の初期構成を作る
5. サービス名は `tdev-demo03-` 接頭辞で統一する

**完了条件**

- ローカルで `web` と `db` の起動手順が定義されている
- Django が PostgreSQL を使う前提で設定されている
- 開発時の最小ディレクトリ構成が確定している

### Step 2. アカウントと認証の受け皿を作る

**目的:** ポータル認証引継ぎを受けるための最低限のアプリ基盤を用意する。

**やること**

1. `accounts` アプリを追加する
2. 利用者、ロール、セッション補助に必要なモデル方針を固める
3. ポータル連携を前提にした middleware または session 初期化方針を定義する
4. アプリ単独ログイン画面を作らない前提で未認証時の扱いを整理する

**完了条件**

- 認証の責務が Django サーバー側に置かれている
- `portal_token` をフロントエンドで扱わない方針が実装に反映できる状態になっている
- 認証前提が `doc\spec-auth\prompt.md` と矛盾しない

### Step 3. 資産台帳の基盤を作る

**目的:** IT資産管理デモの中心データを扱えるようにする。

**やること**

1. `assets` アプリを追加する
2. `AssetCategory` と `Asset` のモデルを追加する
3. 資産状態、シリアル番号、管理情報を定義する
4. Django Admin から台帳管理できるようにする
5. 台帳の初期データ投入方針を決める

**完了条件**

- 資産カテゴリと資産台帳を登録できる
- 在庫状態を表現できる
- Admin で基本運用ができる

### Step 4. 資産一覧と詳細画面を作る

**目的:** 利用者向け・管理者向けの参照導線を整える。

**やること**

1. 資産一覧画面を作る
2. 資産詳細画面を作る
3. 検索・絞り込みの最小機能を入れる
4. 利用者向け表示と管理者向け表示の差分を整理する

**完了条件**

- 資産一覧と詳細を Django Templates で表示できる
- 台帳確認の基本導線が成立している

### Step 5. 貸出申請フローを作る

**目的:** 最初のデモ対象ユースケースの前半を成立させる。

**やること**

1. `loans` アプリを追加する
2. `LoanRequest`、`LoanRecord`、`ReturnRecord` のモデル方針を固める
3. 貸出申請フォームを作る
4. 自分の貸出一覧を作る
5. 貸出申請一覧を管理者向けに作る
6. 貸出可否の業務ルールをサーバー側で判定する

**完了条件**

- 社員が貸出申請できる
- 情シスが申請を確認できる
- 貸出不可状態の資産を弾ける

### Step 6. 承認と返却を作る

**目的:** 貸出の状態遷移を最後まで通せるようにする。

**やること**

1. 承認処理を追加する
2. 貸出確定時に資産状態を更新する
3. 返却申請画面を作る
4. 返却確認処理を追加する
5. 状態遷移を専用処理に集約する

**完了条件**

- 申請 → 承認 → 貸出中 → 返却 → 在庫 の流れが成立する
- 不正な状態遷移を防げる
- 主要操作が追跡可能である

### Step 7. インシデントと棚卸しを追加する

**目的:** CRUD だけではない業務アプリらしさを補強する。

**やること**

1. `incidents` アプリで故障、紛失、廃棄を扱う
2. `inventory` アプリで棚卸しセッションと結果入力を扱う
3. 故障中、紛失済み、廃棄済みの資産に対する制約を実装する
4. 棚卸し差異を確認できるようにする

**完了条件**

- インシデントを記録できる
- 棚卸しの基本フローを回せる
- 例外系シナリオをデモで見せられる

### Step 8. 監査ログとデモ仕上げを行う

**目的:** デモとして説明しやすい状態に仕上げる。

**やること**

1. `auditlogs` アプリで主要操作を記録する
2. デモ用シードデータを用意する
3. UI の最低限の調整を行う
4. 実演手順と README 更新対象を整理する

**完了条件**

- 主要な状態変更や承認操作が監査できる
- デモ用データで正常系と例外系を再現できる
- 実演しやすい最小品質に達している
