# Copilot 必須ルール

このファイルは、`demo03_am` で Copilot が**必ず守るべきルール**をまとめたものです。  
作業前に必ず参照させます。

## 1. 基本原則

1. `doc\spec\it-asset-management-demo.md` を読まずに実装を始めない
2. 不明点は既存文書の範囲で判断し、勝手にスコープを広げない
3. 既存方針に反する独自設計を持ち込まない
4. 依頼対象に直接関係しない箇所を勝手に変更しない
5. 一時しのぎではなく、計画書と整合する形で直す

## 2. 技術選定ルール

1. 言語は **Python**
2. Web フレームワークは **Django**
3. DB は **PostgreSQL**
4. 画面は **Django Templates** を基本にし、必要最小限だけ HTMX または JavaScript を使う
5. 管理画面は **Django Admin** を活用する
6. 構成は **Django 中心のモノリシック構成** を崩さない

## 3. 認証ルール

1. tanoshimi.dev ポータルの認証引継ぎを前提にする
2. アプリ単独のログイン画面を新設しない
3. `portal_token` をフロントエンド JavaScript で扱わない
4. 認証と認可の最終判定は Django サーバー側で行う
5. `demo01_crm` と整合しない認証方式へ変更しない

## 4. 実装ルール

1. 業務ルールは Django 側で最終判定する
2. 貸出可否、返却可否、状態遷移、棚卸し差異判定はサーバー側で保証する
3. テンプレートに業務判定を書きすぎず、条件分岐は Python 側に寄せる
4. 状態遷移は ad-hoc に更新せず、メソッドまたは専用処理に集約する
5. Django Admin は管理導線として使うが、利用者向け導線の代替にはしない
6. 重要操作は監査ログに残す前提で設計する

## 5. スコープ意識

1. 主題は **IT資産台帳と貸出管理**
2. 中核機能は **資産台帳、貸出申請、承認、返却、故障/紛失/廃棄、棚卸し、操作履歴**
3. `demo02_ferms` の予約管理前提を持ち込まない
4. 外部 MDM 連携、Active Directory 連携、複雑なワークフローエンジン、モバイルアプリは今回の標準スコープに含めない

## 6. ドキュメントルール

1. 仕様や前提を変える実装では、関連ドキュメントも更新する
2. 実装判断の正本は `README.md` ではなく `doc\spec\it-asset-management-demo.md`
3. 認証関連の前提確認には `doc\spec-auth\prompt.md` を参照する
4. 恒久価値のない一時メモを増やしすぎない

## 7. 変更禁止・注意事項

1. 指示なしで本番運用詳細を README や補助文書へ書き足しすぎない
2. セキュリティ上重要な設定値をソースへ直書きしない
3. 不要なライブラリを勝手に追加しない
4. 未依頼の大規模リファクタリングをしない
5. フロントエンドに認証や業務ルールの責務を移さない

## 8. 作業前チェック

作業開始前に最低限、以下を確認する。

- 今回のタスクに関係する仕様はどこか
- 認証やセッションに影響するか
- 資産台帳、貸出、返却、インシデント、棚卸しのどこに影響するか
- 利用者向け画面と管理画面の両方に影響するか
- ドキュメント更新が必要か
